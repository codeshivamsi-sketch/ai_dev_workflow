description: Generate 6 diagram-only views (mindmap, schema, delete behavior, data access, deployment, request-flow sequence) from codegraph and save them
---
This command produces a diagram-centric reference: six views, each a
section heading + one-line caption + a single Mermaid diagram, no other
explanation text (the one exception is noted below).

a. **Code Structure (HLD).** Query codegraph and generate a Mermaid mindmap
   of this repo's architecture — modules, their responsibilities, and how
   data flows between them. Group semantically (what the code does), not
   just by folder structure.

b. **Database Schema (LLD).** Find the ORM models in the codebase
   (SQLAlchemy/Prisma) and derive a Mermaid erDiagram from them — tables,
   columns, foreign keys, and relationships. Skip this if the codebase has
   no ORM models.

c. **Delete Behavior (LLD).** Still gated on "has ORM models": scan the
   models' FK definitions for delete-behavior rules — SQLAlchemy
   `ondelete=` arguments, or `onDelete` in `schema.prisma`. If none exist,
   skip this section entirely. Otherwise derive a Mermaid flowchart: one
   node per table (same DB cylinder shape as Data Access Map below), edges
   from parent table to child table styled by the rule — solid, labeled
   `CASCADE`, for `ondelete=CASCADE`/`onDelete: Cascade`; dashed, labeled
   `RESTRICT ⛔`, for `ondelete=RESTRICT`/`onDelete: Restrict`. Add a
   one-line legend under the diagram.

d. **Data Access Map (LLD).** Still gated on "has ORM models": trace every
   endpoint in the codebase (not just changed ones) to the ORM
   models/tables it reaches — the same endpoint/table/access edge shape the
   db-blast skill uses (endpoint, table, access R/W/RW), just without a
   new/removed/changed status column since this isn't diff-scoped. Derive a
   Mermaid flowchart from the edge list: one node per endpoint, one node
   per table (DB cylinder shape), edges labeled with the access type.

e. **Deployment (HLD).** Find `docker-compose.yml`/`compose.yaml` in the
   repo. Derive a Mermaid flowchart of the services it defines and how
   they're networked (`depends_on`, shared networks, exposed ports). Skip
   this section if no compose file exists.

f. **Request Flow (LLD).** Via codegraph, find the most representative
   write (POST) endpoint in the codebase — prefer whichever has the
   deepest/most complex call chain if several are close — and derive a
   Mermaid sequenceDiagram of its call flow (client → route →
   service/business logic → ORM/DB → response). Skip this section if the
   codebase has no write endpoints.

Write the result to docs/architecture.md, with a one-line header noting it
was generated by /arch and the date (get the date by running
`date +%Y-%m-%d`, never guess it), then six sections in this order:
`## Code Structure`, `## Database Schema`, `## Delete Behavior`,
`## Data Access Map`, `## Deployment`, `## Request Flow` — apply each
section's skip condition from above (Database Schema and Data Access Map
are omitted together if there are no ORM models; Delete Behavior, Deployment,
and Request Flow are each omitted on their own per their individual
conditions, even when the other ORM sections are present).

Every section is a one-line caption plus its diagram — nothing else. The
one exception is Delete Behavior: if docs/architecture.md already exists,
read it first and preserve any hand-written prose already under that
section (everything after the legend, up to the next `##` heading or end
of file) — that's the maintainer's design rationale (e.g. "why RESTRICT")
and must survive regeneration. Only the diagram + legend above it gets
regenerated. The other five sections are fully regenerated — overwrite
them, this part of the file is never hand-edited.

Then tell me the file path. GitHub and VS Code render the mermaid blocks
natively.
