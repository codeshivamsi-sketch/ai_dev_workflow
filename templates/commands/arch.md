description: Generate the architecture mindmap + schema ER diagram + data access map + delete behavior diagram from codegraph and save them
---
Query codegraph and generate a Mermaid mindmap of this repo's architecture —
modules, their responsibilities, and how data flows between them. Group
semantically (what the code does), not just by folder structure.

Then find the ORM models in the codebase (SQLAlchemy/Prisma) and derive a
Mermaid erDiagram from them — tables, columns, foreign keys, and
relationships. Skip this if the codebase has no ORM models.

Then, still via codegraph and gated on that same "has ORM models" check,
trace every endpoint in the codebase (not just changed ones) to the ORM
models/tables it reaches — the same endpoint/table/access edge shape the
db-blast skill uses (endpoint, table, access R/W/RW), just without a
new/removed/changed status column since this isn't diff-scoped. Derive a
Mermaid flowchart from the edge list: one node per endpoint, one node per
table (DB cylinder shape), edges labeled with the access type.

Then, still gated on "has ORM models", scan the models' FK definitions for
delete-behavior rules — SQLAlchemy `ondelete=` arguments, or `onDelete` in
`schema.prisma`. If none exist, skip this section entirely. Otherwise
derive a Mermaid flowchart: one node per table (same DB cylinder shape as
the Data Access Map), edges from parent table to child table styled by the
rule — solid, labeled `CASCADE`, for `ondelete=CASCADE`/`onDelete: Cascade`;
dashed, labeled `RESTRICT ⛔`, for `ondelete=RESTRICT`/`onDelete: Restrict`.
Add a one-line legend under the diagram.

Write the result to docs/architecture.md, with a one-line header noting it
was generated by /arch and the date (get the date by running
`date +%Y-%m-%d`, never guess it), then four sections: `## Architecture`
with the mindmap, `## Schema` with the erDiagram, `## Data Access Map` with
the endpoint→table flowchart (omit both Schema and Data Access Map if there
are no ORM models), and `## Delete Behavior` with the CASCADE/RESTRICT
flowchart and legend (omit on its own if there are no FK delete rules, even
when the other ORM sections are present).

Architecture, Schema, and Data Access Map are fully regenerated — overwrite
them, this part of the file is never hand-edited. Delete Behavior is
different: if docs/architecture.md already exists, read it first and
preserve any hand-written prose already under that section (everything
after the legend, up to the next `##` heading or end of file) — that's the
maintainer's design rationale (e.g. "why RESTRICT") and must survive
regeneration. Only the diagram + legend above it gets regenerated.

Then tell me the file path. GitHub and VS Code render the mermaid blocks
natively.
