Parse $ARGUMENTS first — this command has three modes:

- **Starts with `show flow `:** the rest is an endpoint (path or handler
  name, e.g. `POST /checkout` or `checkout`). Resolve it via codegraph (if
  more than one endpoint matches, list the candidates and ask which one),
  then trace its full call path hop by hop until a hop adds no new symbols
  (fixpoint — same no-fixed-depth rule /blast uses): the handler → every
  function it calls transitively → every DB read/write it reaches (ORM
  calls → tables, marked READ/WRITE). Render a Mermaid mindmap of that call
  tree — root is the endpoint, branches are the call chain, leaves are the
  DB operations. Show it directly in your response — one-line caption, no
  other prose, nothing written to a file. Stop here; skip everything below.

- **Starts with `sequence `:** same resolve-and-trace as above, but render
  a Mermaid sequenceDiagram instead (which functions talk, in what order).
  Show it directly in your response — one-line caption, no other prose,
  nothing written to a file. Stop here; skip everything below.

- **Otherwise (no arguments, or anything else):** generate the two static
  diagrams below.

a. **High-Level Architecture (HLD).** Discover the system's real components
   from the codebase itself — services/processes defined in
   `docker-compose.yml`/`compose.yaml` (if present), module/service
   boundaries from codegraph, and detected datastores/caches/queues (DB
   from ORM config, Redis/queue clients from imports/dependencies). Derive
   ONE Mermaid graph showing every component found, its ports, and the data
   flows between them (who calls/reads/writes whom). Never invent a
   component that isn't verifiable from the code; skip anything you can't
   confirm.

b. **ER Diagram (LLD).** Find the ORM models in the codebase
   (SQLAlchemy/Prisma) and derive a Mermaid erDiagram from them — tables,
   columns, foreign keys, and relationships. Skip this entirely if the
   codebase has no ORM models. Fold delete behavior into this same diagram
   instead of a separate section: for every FK that has a delete rule
   (SQLAlchemy `ondelete=`, or `onDelete` in `schema.prisma`), append it to
   that relationship's label — e.g. `AUTHORS ||--o{ BOOKS : "author_id,
   CASCADE"` or `"book_id, RESTRICT ⛔"`. Leave FKs with no explicit delete
   rule unlabeled for that part.

Write the result to docs/architecture.md (fully regenerated — overwrite it,
never hand-edited), with a one-line header noting it was generated by
/arch and the date (get the date by running `date +%Y-%m-%d`, never guess
it), then two sections in this order: `## High-Level Architecture`,
`## ER Diagram` (omit if there are no ORM models). Each section is a
one-line caption plus its diagram — nothing else.

Close the file with a two-line note: "For a single endpoint's call trace,
run `/arch show flow [endpoint]` (mindmap of calls). For its sequence
diagram, run `/arch sequence [endpoint]`." Neither of those is stored in
this file — they're shown inline, on demand.

Then tell me the file path. GitHub and VS Code render the mermaid blocks
natively.
